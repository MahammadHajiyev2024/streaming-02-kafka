## Custom Project

### Dataset
The dataset utilized by the Kafka producer is a simulated real-time retail transaction log.
- **Dataset File Name:** `data/sales.csv`.
- **Record Types:** It contains e-commerce and retail sales records representing individual consumer orders.
- **Included Fields:** Each record contains fields such as `order_id`, `datetime`, `region_id`, `currency_code`, `product_id`, `unit_price`, `quantity`, `is_online`, `customer_id`, `device_type`, and `payment_method`.
- **Modifications:** The original sales dataset structure was kept intact to simulate a standardized enterprise data source.

### Kafka Messages
- **Producer Output:** The producer reads rows from the CSV file, transforms them into JSON strings, and publishes them as live events.
- **Kafka Topic:** The topic used for streaming messages is `streaming-02-kafka-case`.
- **Message Key:** The `region_id` field (e.g., `US-TX`, `CA-QC`) is explicitly extracted and applied as the Kafka message key. This ensures that all messages from the same geographic region map to consistent processing boundaries if scaled.
- **Field Changes:** No fields were altered during transmission; the entire structured transaction record is safely enveloped into the message payload.

### Consumer Processing
- **Consumer Input:** The script `kafka_consumer_hajiyev_aggregator.py` subscribes to the topic and pulls the raw JSON sales dictionaries.
- **Message Budget:** It streams messages continuously up to a maximum limit configured by `CONSUMER_MAX_MESSAGES` (default 1000) or terminates early if a 10-second timeout occurs without receiving new data.
- **Logging Behaviour:** For every message pulled, it logs the raw content, tracks ingestion success, and prints real-time operational state metrics like: `Updated region=US-TX, product=PY-DATA-002, revenue=49.99`.
- **CSV Output Persistence:** The aggregated multi-dimensional metrics are dumped directly into `data/output/consumed_sales_hajiyev_summary.csv`.
- **Specific Processing and Filtering Logic:** The consumer filters and focuses explicitly on the `region_id`, `product_id`, `unit_price`, and `quantity` fields. Instead of logging rows side-by-side, it processes the metrics through two memory dictionaries (`region_stats` and `product_stats`) to compute:
  - Cumulative transactional revenue per category (`price * quantity`).
  - Total volume item units ordered.
  - Calculated mathematical averages for both order revenue size and basket item depth.

### Experiments
- **Phase 4 Modification:** Modified terminal run structures to use explicit module discovery (`uv run python -m streaming.kafka_consumer_hajiyev_aggregator`) to prevent local directory imports from breaking during execution.
- **Phase 5 Application:** Created a dual-aggregation loop logic mechanism. The script was decoupled so that a single stream event updates both a geographic performance dataset and an inventory demand dataset simultaneously before storing the data in a standardized row-type flag structure.

### Results
When launching Kafka, running the producer, and spinning up the aggregator consumer, the pipeline successfully executed end-to-end. The console actively logged messages as they were produced, and the aggregator read every payload immediately without data loss. The consumer cleanly exited upon timeout, saving the grouped summary tables to `consumed_sales_hajiyev_summary.csv`.

### Interpretation
- **Changes from Example:** The base consumer example only logged individual rows to a flat historical record. This project transformed that behavior into live analytic processing, ensuring data is grouped into actionable summaries before it hits permanent storage.
- **Core Streaming Takeaway:** Observing messages move highlighted the decoupling advantage of Kafka. The producer can run at full throttle, and multiple analytical consumers can attach or detach from the stream independently without impacting the core transaction engine.
- **Value to a Business:** A business organization can use this stream to see live regional financial velocity and inventory item drain. Instead of waiting for batch reporting at the end of a fiscal quarter, the organization can monitor performance in real time.
- **Business Intelligence Gained:** The aggregated records give instant visibility into which geographic locations yield the highest revenue density and which specific items maintain the highest demand velocity. This provides the exact information needed for automated supply chain restocking or targeted geographic marketing campaigns.
